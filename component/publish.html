修改版本自增版本号 ​

遵循发布的版本号的规则 ​

npm版本号遵循语义版本控制规则，由三个部分（主版本号、次版本号、修订版本号）组成

  • 主版本号（Major）。当项目进行重大变更，引入了不兼容的修改时，主版本号会递增。
  • 次版本号（Minor）。当项目添加新功能或对现有功能进行兼容性修改时，次版本号递增。
  • 修订版本号（Patch）。用于修复bug或进行向后兼容的修改，修订版本号会递增

此外，版本号的比较和匹配规则是基于这些版本号的递增关系，常用的版本运算符如波浪号（~）、插入符号（^）等，用于指定依赖的版本范围。 例如，~version 会匹配某个特定版本，并允许次要版本和修订版本的更新；^version 则会匹配某个特定版本的次要版本和修订版本，但主要版本可以更新

版本发布规则 ​

内测版本(alpha)【发布测试或者开发版本】 ​

  • alpha alpha 是内部测试版，由于功能亦未完善，一般不向外发布，会有很多 Bug，除非你也是测试人员，否则不建议使用
  • 发布规则：1.0.0 -> 1.0.1-alpha -> 11.0.1-alpha.1 -> 1.0.1-alpha.2

beta ​

  • 该版本相对于 α 版已有了很大的改进，消除了严重的错误，但还是存在着缺陷，需要经过多次测试来进一步消除
  • bate 版本最早对外公开的软件版本，会收集使用者的反馈意见，以便作为进一步修正软件的参考，这个阶段的版本会一直加入新的功能和特性。
  • 发布规则：1.0.1-beta -> 1.0.1-beta.1 -> 1.0.1-beta.2 -> 1.0.1-beta.3

标准版本（release） ​

  • 发布规则：1.0.0 -> 1.0.1 -> 1.0.2

发布 ​

bash
 npm publish --registry https://repo-fe.nanshe-arch.com

或者

bash
 npm publish
